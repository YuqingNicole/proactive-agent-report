Proactive Agent 全网调研报告

从 Reactive 到 Proactive：实现方式全景扫描

📅 2026-03-23 🔍 来源：GitHub / arXiv / Medium / LinkedIn / Salesforce Docs 📊 覆盖 2024–2026 最新进展

TL;DR

Proactive Agent = 持续运行的 Agent，主动在合适时机采取行动，不等用户 prompt。
核心问题有三个：① 何时触发（Trigger）、② 基于什么上下文决策（Context）、③ 采取什么行动（Action）。
目前业界主流实现分为 5 类，从最简单的定时轮询到最复杂的认知架构 Inner Thoughts，成熟度和复杂度差异很大。没有最好的方案，只有最适合场景的方案。

🎯

Reactive vs Proactive：核心区别

为什么这件事难

😴 Reactive Agent（传统）

等待用户 prompt → 响应。生命周期是单次 session，不持续运行。知道你问了什么，不知道你需要什么。ChatGPT、Copilot 都是这个模型。

🚀 Proactive Agent（目标）

持续监控信号 → 自主判断时机 → 主动行动或建议。不需要用户先开口。核心是主动性（Initiative）和时机感（Timing）。OpenClaw、GitHub Copilot Agent Mode 是典型实践。

💡
关键洞察：Proactive Agent 的难点不是 LLM 能力，而是激活机制（Activation）——什么时候叫醒 Agent、用什么上下文、叫醒后做什么。这三个问题决定了一切架构选择。
⚙️

5 种实现方式

从简到复杂，覆盖 2024–2026 主流方案
方式 01

⏰ Schedule-Based（定时触发）

最简单也最常见的方式。用 cron、APScheduler、asyncio 等定时调用 Agent，Agent 在每次触发时扫描上下文、判断是否需要行动。

典型实现：每天 23:00 扫描日志 → 提取关键观察写入 OBSERVATIONS.md（就是本项目的 observer.py）。也是 grapeot context-infrastructure 的核心机制。

cron APScheduler asyncio loop Python 最易实现

✓ 可预测、可审计
✓ 无需事件基础设施
✗ 时机不精准（可能太早/太晚）
方式 02

📡 Event-Driven（事件驱动触发）

当特定事件发生时才触发 Agent，而不是轮询。事件可以来自：Webhook、数据库 CDC（Change Data Capture）、消息队列（Kafka/RabbitMQ）、Platform Events（Salesforce）等。

典型实现：GitHub Issue 被创建 → Webhook → Agent 自动分析并回复。新邮件到达 → 邮件钩子 → Agent 分类处理。Salesforce Agentforce 是企业级典型案例。

Webhook CDC Kafka Platform Events n8n Zapier

✓ 响应及时、不浪费算力
✓ 天然适合高频变化场景
✗ 需要事件基础设施
✗ 不适合需要主动扫描的场景
方式 03

🔄 Hybrid：Schedule + Event（混合触发）

业界最推荐的方式：定时触发（Proactive）+ 事件触发（Reactive）双轨并行。

事件触发让 Agent 对重要变化即时响应（新邮件、报警、部署），定时触发保证定期健康检查、日志审计、安全扫描。两者不是竞争关系，是互补的。

Montana Flynn（LinkedIn，2026-03）总结：「Reactive triggers make agents timely. Proactive triggers make agents reliable.」

双轨并行 互补设计 生产推荐
方式 04

🧠 Context-Aware Timing（上下文感知触发）

Agent 自己决定「什么时候说话」。通过多因子决策引擎（对话流、用户参与度、紧急程度、响应间隔）动态计算下次响应时机，而不是固定间隔。

leomariga/ProactiveAgent（Python 库）的核心设计：min_response_interval、max_response_interval、probability_weight 共同决定激活时机。类似人类在群聊里「感觉到合适时机才插话」的行为。

多因子决策 动态 sleep leomariga/ProactiveAgent 适合对话场景
方式 05

🧬 Inner Thoughts（认知架构触发）

最前沿的研究方向（CHI 2025 论文：Proactive Conversational Agents with Inner Thoughts）。Agent 像人类一样，在听对话的过程中并行运行内心思维，形成想法、评估是否值得发言、判断是否需要打断。

五阶段框架：Trigger → Retrieval → Thought Formation → Evaluation → Participation
关键参数：interruptThreshold（1-5 打断阈值）——Agent 多强烈的想法才会打断用户。

通用架构：4 层模型

Manu Francis（Medium, 2025） / 业界共识
① Trigger Layer
→
cron / webhook / event / context-aware timer
↓ 唤醒 Agent
② Sensor / Context Collector
→
抓取信号：日志、邮件、数据库、API、对话历史
↓ 传入上下文
③ Reasoning Layer (LLM)
→
判断信号重要性 → 决定行动类型 → 生成响应
↓ 决策输出
④ Action Layer
→
发通知 / 写文件 / 调 API / 更新数据库 / 触发工作流
↓ （可选）Human-in-the-Loop
⑤ Memory Layer
→
存储历史、状态、用户偏好 → 下次触发时提供上下文
🔁
Memory 是 Proactive Agent 的核心护城河。 没有跨 session 记忆，Agent 每次唤醒都是「失忆」状态，无法积累用户习惯和偏好。MemGPT（2024）、grapeot context-infrastructure、本项目的 observer/reflector 设计，本质上都是在解决这个问题。

3 个关键设计决策

做 Proactive Agent 绕不开的问题

① 触发阈值（When to Fire）

Agent 不能什么都管，否则会「烦死人」。需要明确：什么情况下主动行动，什么情况下保持沉默。
设计方式：固定规则（if 条件满足 then 触发）/ LLM 自主判断（把上下文传给模型让它决定）/ 多因子评分（relevance score > threshold）。
OpenClaw 的 HEARTBEAT.md 设计是一个实用的「沉默规则」实现。

② Human-in-the-Loop（主动行动 vs 建议）

完全自主行动风险高，但每次都等确认效率低。业界共识：低风险行动（写文件、发通知）自主执行；高风险行动（发邮件、删数据、外部 API 写入）需要人工确认。
实现方式：Telegram 按钮确认、Slack Approve / Reject / Skip 三选项。

③ 记忆架构（Memory）

Proactive Agent 需要跨 session 记忆才能真正做到「懂你」。没有记忆，每次唤醒都是从零开始。
分层设计是共识：短期（Working Memory = 当前 context window）→ 中期（OBSERVATIONS.md）→ 长期（rules / axioms / MEMORY.md）。
2026-03 最新 arXiv 论文综述（arxiv:2603.07670）把 memory 总结为 write-manage-read 三步循环，是目前最权威的理论框架。

规律总结：个人场景（OpenClaw/Lindy）以 Schedule + Event 混合触发为主，偏重持久记忆和多渠道集成；企业场景（Salesforce/AutoGen）以事件驱动为主，偏重编排精度和合规性；开发者工具（LangGraph/CrewAI）以框架能力为主，偏重灵活性和性能。
没有哪个产品全能——选择取决于你的触发源是什么，以及谁来消费 Agent 的输出。

Apify 作为信号层：是什么意思？

Proactive Agent 的「眼睛」
💡
核心问题：Agent 怎么知道外部发生了什么？
Proactive Agent 的第一层是信号采集——Agent 需要感知外部变化才能主动行动。Apify 提供现成的「Actor」（爬虫模块），作为 Agent 的感知神经系统。

📍 Apify 在架构里的位置

Apify（信号采集）
   ↓ 发现变化（新文章/价格/提及）
Agent 推理层（这个信号重要吗？）
   ↓ 判断需要行动
Action Layer（通知/写文件/调 API）

🎯 具体能监控什么

RSS Feed Reader — 竞品博客/新闻有新内容
Social Media Monitor — Twitter/Reddit 关键词提及
Web Scraper — 价格变化、产品页更新
Google Search Scraper — 搜索结果排名变化

实际场景举例

Sitesfy 桌游出海：Apify 每天抓欧洲展会参展商名单变化 → 发现新目标公司 → Agent 自动生成个性化开发信
内容运营：监控竞品发了什么文章 → Agent 自动生成选题建议并推送给你
投资监控：抓取特定公司的新闻/招聘动态 → 出现异常信号时 Agent 主动提醒

本质：让 Agent 有眼睛看世界，而不是等人告诉它发生了什么。没有信号层，Proactive Agent 只能靠定时轮询盲跑。

企业级实践：Salesforce Agentforce 解剖

最完整的 Event-Driven Proactive Agent 架构案例

三层架构：Senses + Orchestration + Brain

① Senses（感官） = Platform Events + CDC：数据库变化自动推送给 Agent，不轮询。
② Orchestrator（编排） = Salesforce Flow：提供确定性逻辑和安全护栏，决定什么时候让 AI 介入。
③ Brain（大脑） = Invocable Apex + Agentforce Reasoning：复杂分析 + AI 推理。

核心洞察：Flow 提供确定性，AI 提供灵活性。两者结合才能既安全又智能。 不要让 AI 直接处理退款流程，而是让 AI 决策、Flow 执行。

⚠️
生产警告：「工作在 demo 上成功」不是发布策略。Agentix Labs 总结：tool-calling agent 的生产失败通常很无聊但代价昂贵——错误工具选择、参数错误、重试循环消耗算力、静默数据损坏。必须在发布前建立评估门槛（6 维度：任务成功率、工具正确率、基础性、安全合规、延迟可靠性、单次任务成本）。

🎯
核心判断：Proactive Agent 的价值不在于「技术有多复杂」，而在于「减少了多少用户的认知负担」。从用户视角出发（K01 认知不对称），优先让 Agent 处理你最不想思考的重复判断，而不是追求最先进的架构。
