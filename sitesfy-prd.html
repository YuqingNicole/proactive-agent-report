产品愿景与定位

为什么是 Sitesfy
🎯
核心命题：「你的第一个出海市场情报官」
不是帮你发邮件，是帮你比竞争对手多看一层市场信号——信号差距才是护城河。

❌ 我们不是

邮件群发工具 / SEO 写作工具 / 通用 AI Agent
「帮你执行」的工具已经够多了。

✅ 我们是

出海获客的判断引擎。核心价值在于：把市场信号转化为用户的具体行动指令，让每一个动作都发生在正确的时机、正确的对象上。

Layer 1
Signal Intelligence · 信号情报
监控市场信号：谁在找替代方案、竞品出了什么问题、行业采购信号是什么
输出：判断依据，不是名单
Layer 2
Judgment Engine · 判断引擎
把信号转化为行动指令：「这周德国有3家在问ESG认证」「律所有2家高意图访客」
输出：具体行动指令
Layer 3
Execution Autopilot · 执行
判断正确后才自动化：发邮件、写文章、跟进序列。大量外包给 Instantly/Apollo
Sitesfy 只赚「判断正确」的钱

⚙️ 实现难度标注：哪些地方最难做？

🔴 高难
Proactive Core 的决策逻辑 · 模块 00
核心难点不是技术，是「什么时候该自动执行，什么时候该让人判断」这个边界极难定义。 阈值设太高 → 引擎不动，用户感受不到价值；设太低 → 乱发邮件，砸域名声誉。 需要在真实用户数据中反复标定，MVP 阶段很容易出错。
💡 建议：Phase 1 先硬编码简单规则（回复率＜2% 暂停，＞8% 扩量），不要过早做通用决策引擎
🔴 高难
邮件个性化质量 vs 规模 · Content Engine
LLM 生成的开发信越来越容易被 AI 检测工具识别（ESP 已开始过滤）。 真正有效的个性化需要公司级别的上下文（融资信息、近期动态、职位变动）， 而这些数据获取成本高、时效性强、很难自动化到足够质量。
💡 建议：用「模板+变量」结构而非完全 LLM 生成，关键个性化字段（最近一篇博文/近期招聘信号）由 scraper 实时抓取
🔴 高难
访客意图评分标定 · Capture Engine
每个行业、每个产品的「高意图行为」定义完全不同。 律所访客看「GDPR 罚款案例」页面和看「关于我们」页面的意图差异， 需要人工标注样本才能训练出有意义的模型——而早期根本没有样本数据。
💡 建议：Phase 1 用硬规则替代模型（停留＞3分钟 + 访问定价页 = 高意图），积累 200+ 样本后再引入评分模型
🟡 中难
域名预热自动化管理 · Outreach Engine
预热是机械性工作（每天发 X 封，发件人要变化，内容要真实）， 但跨多个客户域名的并行管理、预热进度追踪、声誉监控，工程量不小。 Instantly 提供了预热功能，但 API 层的精细控制有限。
💡 建议：直接用 Instantly 的内置预热功能，不要自建。Sitesfy 层只做状态监控，不碰预热逻辑
🟡 中难
SEO 文章自动发布到客户网站 · Content Engine
WordPress API 存在版本差异大、认证方式不统一（REST API / XML-RPC / 插件认证）的问题。 Headless CMS 客户更复杂，需要单独适配 Contentful / Sanity / Ghost。 图片处理、分类/标签自动匹配、SEO 插件字段（Yoast/RankMath）填写都是细节陷阱。
💡 建议：Phase 1 只支持 WordPress + Application Password 认证，其他 CMS 全部延后
🟡 中难
Learning Engine 的飞轮打通 · 模块 05
数据反哺 ICP 修正听起来简单，但实际上需要：足够的样本量（＞200 封回复）、 因果判断（是 ICP 的问题还是话术问题？）、防止过拟合（不能因为一个方向回复率高就全押）。 这是一个需要持续迭代的数据工程问题，不是写一次就好的功能。
💡 建议：Phase 1 只做数据采集，不做自动修正。Phase 2 再引入建议层（「建议扩大德国市场比例」），用户确认后才执行
🟡 中难
德语/多语言开发信质量 · Content Engine
LLM 在德语商务邮件上的质量明显低于英语，尤其是德国商业文化中对正式程度的把握（Sie vs du）、 行业术语、以及「不像机器翻译」这个关键要求，很难通过 prompt 完全解决。
💡 建议：用 Claude Opus 4.5 生成，加入德语 native speaker 写的 few-shot 示例，邮件发出前增加「语言质量」自动评分层

技术底层难点

不在功能列表里但决定成败的工程挑战

🔴 实时信号的时效性 vs 处理成本

Proactive Core 需要「实时」感知信号，但真正的实时处理（毫秒级）成本极高。 邮件回复率每 5 分钟轮询一次已经够快，但访客意图识别需要接近实时（用户还在页面上时触达才有意义）。 两种截然不同的延迟需求，共享同一套架构，会产生资源分配和优先级冲突。

解决思路：双轨架构——高频信号（访客）走 Supabase Realtime WebSocket，低频信号（邮件回复）走 cron 轮询。Proactive Core 订阅两个队列。

🔴 多客户隔离 + 数据安全

桌游公司和律所的数据必须严格隔离。LLM 调用时如果不小心把 A 公司的 ICP 数据混入 B 公司的 prompt， 轻则输出错误，重则泄漏商业机密。Supabase 的 Row Level Security 在单表层面好控制， 但跨表 join + LLM context 组装时的隔离非常容易出错。

解决思路：每次 LLM 调用前，在 middleware 层强制注入 tenant_id 过滤，所有数据查询必须带 tenant 上下文。定期审计 LLM prompt 里的数据来源。

🔴 邮件送达率的不可观测性

Instantly 能给你打开率和回复率，但无法告诉你有多少邮件静默进了垃圾桶（既没退信也没显示未送达）。 这意味着 Proactive Core 基于「低回复率」做的判断，可能是数据失真——不是 ICP 错了，是邮件根本没被看到。 两种情况需要截然不同的应对策略，却呈现同样的外部表现。

解决思路：在 Learning Engine 加「送达质量评分」层——监控退信率、DMARC 报告、Spam complaint rate，作为回复率数据的可信度权重。

🟡 LLM 输出的幂等性

同一个用户 URL 和 ICP 条件，两次调用 LLM 可能生成完全不同的开发信内容和关键词列表。 这对「A/B 测试主题行」有价值，但对「修改后重新生成」是灾难——用户无法判断是他的改动生效了， 还是 LLM 随机产生了变化。

解决思路：固定 seed（对支持的模型）+ 版本化存储所有生成结果，修改走 diff 展示，不是整体重新生成。

🟡 Apollo 数据质量参差不齐

Apollo 的联系人数据准确率在不同行业差异很大。欧洲小型桌游零售商的决策人邮箱， Apollo 经常是空的或过期的（人员流动、公司规模小）。 发到错误邮箱不只是浪费，还会拉低域名声誉。

解决思路：接入 Hunter.io 或 Proxycurl 做邮箱验证层，发送前必须通过 MX 记录验证。对验证失败的联系人标记，不纳入发送队列。

🟡 WordPress API 版本碎片化

WordPress 5.x 和 6.x 的 REST API 行为有差异，加上 WooCommerce / Elementor 等插件可能修改默认路由， 导致「自动发布文章」在不同客户网站上行为不一致。Application Password 认证在部分安全插件下会被拦截。

解决思路：发布前先调 /wp/v2/posts?status=draft 做连通性测试，记录每个客户的 WordPress 环境指纹，异常时降级为「生成草稿 + 邮件通知人工发布」。

风险与护城河

核心挑战 + 差异化来源

⚠️ 核心风险

📧
邮件送达率下降：批量个性化邮件的 AI 味越来越容易被 ESP 识别，域名声誉是消耗品
📊
ICP 偏差放大：判断错误 + 自动化 = 规模化地烧钱，早期校准至关重要
🔗
第三方依赖风险：Apollo/Instantly/RB2B 任一定价调整或 API 变更都影响核心功能
⚖️
合规风险：GDPR 对欧洲冷邮件有严格限制，德国市场尤其敏感

🏰 护城河来源

📈
数据飞轮：客户越多 → 行业信号越丰富 → 判断越准 → 回复率越高。跨客户的行业级信号是最大护城河
🎯
时机评分算法：「什么时候发、发给谁」的判断逻辑沉淀在系统里，竞品无法直接复制
🔄
客户上下文积累：使用越久，Sitesfy 越懂这个客户的市场，切换成本越高
💡
按结果定价：长期目标是按有效回复收费，倒逼产品必须真正解决判断问题
