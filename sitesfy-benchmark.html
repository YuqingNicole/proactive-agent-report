Agent Benchmark 框架

多 Agent 工作流质量评估体系：任务拆分 → 执行 → 审核

版本 v1.0 日期 2026-03-23 仓库 botearn/agent-benchmark
↗ GitHub: botearn/agent-benchmark

系统概述

为什么需要 benchmark
💡
三个核心设计原则：
① 可验证的拆分 — 每个子任务有且仅有一句话接受标准
② 失败优先的执行 — 每个任务默认会失败，设计的是失败后如何恢复
③ 嵌入式审核 — 每步完成后立刻 audit，不让错误向下传播
三层工作流
输入任务
→
L1 Decomposer
→
拆分成可执行原子单元（每个有接受标准）
↓
L2 Executor
→
按依赖顺序执行，失败分类处理（重试/降级/人工）
↓
L3 Auditor
→
6维度评分，PASS / SOFT_FAIL / HARD_FAIL / BLOCK
↓
输出 + Benchmark 报告

L1 任务拆分（Decomposer）

好的拆分 = 可验证性

拆分判断标准：一个问题

「你能不能用一句话描述它完成了算什么？」
不能 → 继续拆。能 → 停止。
这解决了过拆（20个步骤做2步的事）和欠拆（一个任务有3个目标）两个方向的问题。

Sequential Chain

输出 N 是输入 N+1 的前提。适合：Research → Draft → Edit

T → [T1 → T2 → T3]

Parallel Fan-out

子任务互相独立，并发执行。适合：同时分析3个竞品

T → [T1, T2, T3] → Merge

Difficulty-Adaptive

先估算复杂度，再路由到简单/复杂流水线

T → [难度评估] → Simple / Complex

Subtask Schema（每个子任务必须声明的字段）

{
  "id": "T1",
  "title": "...",
  "description": "...",
  "acceptance_criteria": "如何验证完成",  // 最重要
  "agent_role": "researcher|writer|coder|analyst|reviewer",
  "depends_on": [],          // 依赖关系
  "can_parallelize": false,  // 是否可并行
  "timeout_s": 60,
  "retry_limit": 2
}
常见拆分失败症状修复
过度拆分20个步骤做2步的事合并总是同时出现的原子单元
欠拆分一个子任务有3个不同目标按接受标准拆，不能一句话描述就继续拆
缺依赖声明步骤2在步骤1完成前就开始显式声明 depends_on 字段
上下文污染Agent 携带不相关的前序上下文严格限定每个子任务的输入范围

L2 执行（Executor）

失败优先设计
🔧
失败分类（不同失败类型对应不同恢复策略）： Tool failure（重试）/ LLM failure（换模型）/ Context failure（补充输入）/ Logic failure（人工介入）/ Audit rejection（带 feedback 重试）
Agent Role职责典型工具
researcher信息采集web_search, read_url
writer结构化文本生成LLM generation
coder代码编写/运行code_exec, file_write
analyst数据处理与综合data_tools, LLM
reviewer质量检查LLM judge
orchestrator任务分解和路由decomposition logic

L3 质量审核（Auditor）

6维度评分 + 决策树
维度权重衡量内容
Task Completion30%是否实现了目标？
Accuracy25%内容是否真实，无幻觉？
Format Adherence15%输出格式是否符合声明？
Conciseness10%无多余填充内容？
Tool Use10%工具选择和参数是否正确？
Safety10%无违规内容？

审核决策树

PASS score ≥ 70 且 safety ≥ 7 → 接受，进入下一步
SOFT_FAIL score 50-69 → 带 feedback 重试同一 agent
HARD_FAIL score < 50 → 升级到人工 或 换 agent
BLOCK safety < 5 → 立即停止，告警人工

Benchmark 评分指标

系统级健康度追踪
指标计算方式目标值
Task Success Rate通过任务 / 总任务> 80%
Subtask Pass Rate通过子任务 / 总子任务> 85%
First-attempt Pass Rate不重试就通过 / 总子任务> 70%
Retry Effectiveness重试后通过 / 总重试次数> 60%
Human Escalation Rate升级人工 / 总任务< 5%
Avg Latency (simple)简单任务平均耗时< 10s

Test Case 覆盖类型

⭐ simple_linear — 3步顺序任务
⭐⭐ parallel_merge — 3条并行任务+合并
⭐⭐ dependency_chain — 5步依赖链
⭐⭐⭐ failure_recovery — 强制工具失败，测试恢复能力
⭐⭐⭐ long_horizon — 10步+分支任务
⭐⭐⭐⭐ adversarial — 模糊输入+冲突需求
